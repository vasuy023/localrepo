================================================================================
                          GIT BASICS & WORKFLOW GUIDE
================================================================================

WHAT IS GIT?
Git is a version control system that tracks changes to files and allows collaboration.
It helps you:
- Track file history
- Revert to previous versions
- Work on features separately (branches)
- Collaborate with others

================================================================================
                        BASIC WORKFLOW (MOST IMPORTANT!)
================================================================================

Step 1: MAKE CHANGES
- Edit files in your project

Step 2: STAGE CHANGES
git add <filename>      # Add specific file
git add .              # Add all files

Step 3: COMMIT CHANGES
git commit -m "descriptive message"

Step 4: PUSH TO GITHUB
git push

IMPORTANT: If a file is already committed and hasn't changed, git add won't 
stage it. You need to MODIFY THE FILE FIRST.

================================================================================
                        IMPORTANT GIT CONCEPTS
================================================================================

BRANCHES:
- main: Your production/primary branch
- feature1, feature2, etc: Branches for new features
- Branches allow parallel development

Check branches:
git branch -a          # See all branches
git branch             # See local branches

MERGING BRANCHES:
git merge <branch-name>     # Merge a branch into current branch
After merge, files from the feature branch are combined into main.

================================================================================
                          KEY GIT COMMANDS
================================================================================

Command                          What it does
================================================================================
git status                       Check what files changed
git add <file>                   Stage file for commit
git add .                        Stage all files
git commit -m "msg"              Create a commit with message
git push                         Upload commits to GitHub
git pull                         Download changes from GitHub
git log --oneline                See commit history
git branch -a                    See all branches
git branch <name>                Create new branch
git checkout <branch>            Switch to branch
git merge <branch>               Merge a branch
git diff <branch1> <branch2>     Compare branches
git show <commit-id>             See details of a commit
git restore <filename>           Undo changes in file
git restore --staged <file>      Unstage a file

================================================================================
                            IMPORTANT RULES
================================================================================

DO:
✓ Write CLEAR commit messages (e.g., "Add button feature", "Fix README typo")
✓ Commit frequently with logical changes
✓ Always push your work to GitHub
✓ Use branches for new features
✓ Pull before starting work if collaborating
✓ Review changes before committing (git status, git diff)
✓ SAVE FILES before running git add

DON'T:
✗ Commit without messages
✗ Mix multiple unrelated changes in one commit
✗ Forget to push your work
✗ Work directly on main for big features (use branches)
✗ Edit files and expect them to auto-stage
✗ Leave uncommitted changes for long periods

================================================================================
                         WORKFLOW SUMMARY
================================================================================

1. CLONE/CREATE repository
   git clone <url>    # Copy remote repo

2. CREATE feature branch (optional but recommended)
   git branch <feature-name>
   git checkout <feature-name>

3. MAKE CHANGES → STAGE → COMMIT → PUSH
   git add .
   git commit -m "message"
   git push

4. MERGE to main (when feature is ready)
   git checkout main
   git merge <feature-branch>
   git push

================================================================================
                       GITHUB INTEGRATION
================================================================================

Local repo = Your computer
Remote repo = GitHub (origin)
git push = Send changes to GitHub
git pull = Get changes from GitHub
Pull Request = Request to merge feature branch to main (on GitHub)

================================================================================
                       COMMON SCENARIOS
================================================================================

SCENARIO: File doesn't stage
Problem: git status shows clean but you edited a file
Solution: Make sure you SAVED THE FILE (Ctrl+S), then git add

SCENARIO: Merged feature to main
Result: Files from feature branch are now in main, no separate folder needed
Check: git log --oneline --graph --all shows merge

SCENARIO: Want to undo changes
git restore <filename>        # Undo changes in file
git restore --staged <file>   # Unstage a file

================================================================================
                   RESOLVING MERGE CONFLICTS
================================================================================

WHAT IS A MERGE CONFLICT?
A merge conflict happens when Git can't automatically combine changes from two
branches because they modified the same lines in the same file.

WHEN DOES IT HAPPEN?
Scenario:
- You on "main" edit line 5 of index.html
- Someone on "feature1" also edits line 5 differently
- You try to merge feature1 into main
- Git says: "I don't know which version to use!"

HOW TO IDENTIFY A CONFLICT:
Run:
git status

Shows:
Both modified:   index.html

HOW TO RESOLVE IT:

Step 1: Open the conflicted file
Git marks conflicts like this:

<<<<<<< HEAD
<p>This is version A</p>
=======
<p>This is version B</p>
>>>>>>> feature1

Step 2: Choose which version to keep
Three options:

Option A: Keep your main version
<p>This is version A</p>

Option B: Keep feature1 version
<p>This is version B</p>

Option C: Keep BOTH versions
<p>This is version A</p>
<p>This is version B</p>

Step 3: Remove the conflict markers
Delete these lines completely:
<<<<<<< HEAD
=======
>>>>>>> feature1

Step 4: Stage, commit, and push
git add index.html
git commit -m "Resolve merge conflict in index.html"
git push

COMPLETE EXAMPLE:
=================

# Try to merge
git merge feature1

# Git says: CONFLICT!
# Check status
git status

# Edit the file in your editor and choose versions
# Remove the conflict markers

# Stage the fixed file
git add index.html

# Commit the merge
git commit -m "Resolve merge conflict"

# Push to GitHub
git push

BEST PRACTICES TO AVOID CONFLICTS:
===================================
1. COMMUNICATE - Talk to team about what you're editing
2. KEEP BRANCHES SHORT - Merge soon, don't work too long separately
3. PULL FREQUENTLY - Stay updated with main branch
4. EDIT DIFFERENT FILES - When possible, different team members edit different files
5. USE BRANCHES - Don't all work on main

QUICK CONFLICT SUMMARY:
=======================
Problem         → Git can't merge automatically
Identify        → git status shows conflicts
Find conflict   → Look for <<<<<<, ======, >>>>>> markers
Choose version  → Keep the code you want
Remove markers  → Delete <<<, ====, >>> lines
Complete        → git add → git commit → git push

COMMON COMMANDS FOR CONFLICTS:
==============================
git status                  # See which files have conflicts
git diff                    # See the differences
git merge --abort          # Cancel merge if you want to start over
git rebase --abort         # Cancel rebase if you started one

IMPORTANT:
Only you can decide which code is correct. Git can't know - you must choose!
